# 短视频多平台标题描述生成器 — 设计文档

## 一、产品概述

一个浏览器扩展插件，用户上传短视频后，自动提取视频中的语音内容和关键画面，调用大模型（DeepSeek / 豆包 / Kimi / OpenAI 等）生成适配各平台调性的标题、描述和标签，支持一键复制到抖音、小红书、快手、B站、视频号、YouTube Shorts 等平台。

**核心价值**：把「看视频 → 想标题 → 改描述 → 适配平台」这套 10-20 分钟的重复劳动，压缩到 30 秒内完成。

---

## 二、核心功能清单

### 2.1 视频导入
- 拖拽 / 点击上传短视频（支持 mp4 / mov / webm）
- 粘贴视频 URL（从剪贴板读取）
- 视频预览播放器（内嵌在 popup 中）

### 2.2 内容提取
- **语音转文字**：调用 Whisper API（或各 LLM 厂商的 ASR 接口）将视频音频转为文本
- **关键帧提取**：每隔 N 秒抽取一帧画面，调用多模态模型理解视觉内容
- **转录文本可编辑**：用户可手动修正转写错误，修正后会重新生成

### 2.3 多平台智能生成
- 选择目标平台组合（多选）
- 根据每个平台的**内容调性、字数限制、标签风格**生成差异化内容
- 每平台生成：标题（2-3 个备选）、描述正文、推荐标签/话题

### 2.4 一键操作
- **单平台复制**：每条内容旁边一个复制按钮
- **全部复制**：按平台分组全部复制到剪贴板
- **导出**：导出为 JSON / CSV 方便批量上传工具使用

### 2.5 设置与配置
- 配置各 LLM 的 API Key 和 Endpoint
- 自定义每平台的 Prompt 模板
- 选择默认目标平台
- 历史记录查看

## 四、详细功能设计

### 4.1 视频处理管线

```
用户上传视频
    │
    ▼
┌──────────────┐    ┌─────────────────┐
│ 提取音频轨道  │───▶│ 发送至 ASR 服务  │───▶ 转写文本
│ (Web Audio   │    │ (Whisper/其他)  │
│  API分离)    │    └─────────────────┘
└──────────────┘
    │
    ▼
┌──────────────┐    ┌─────────────────┐
│ 关键帧提取    │───▶│ 多模态模型分析   │───▶ 视觉描述
│ (Canvas 截帧) │    │ (GPT-4V/豆包)  │
└──────────────┘    └─────────────────┘
    │                       │
    └───────────┬───────────┘
                ▼
    ┌─────────────────────┐
    │ 合并文本 + 视觉信息   │
    │ → 发送给 LLM 生成    │
    └─────────────────────┘
```

**关键帧提取策略**：
- 短视频（≤60s）：每 5 秒抽 1 帧，最多 12 帧
- 中视频（≤5min）：每 10 秒抽 1 帧，最多 30 帧
- 长视频（>5min）：每 20 秒抽 1 帧，最多 30 帧
- 前端用 Canvas 将帧转为 base64 JPEG（质量 0.6），控制总大小 < 5MB

### 4.2 平台适配策略

| 平台 | 标题字数 | 描述风格 | 标签规则 | 特殊要求 |
|------|---------|---------|---------|---------|
| **抖音** | ≤55字 | 口语化、钩子开头、引导互动 | 2-5个热门话题标签 | 强调「前3秒吸引」，标题要有悬念感 |
| **小红书** | ≤20字 | 种草笔记风、详细分点、emoji丰富 | 末尾加主题标签 | 像一篇mini笔记，结构化：标题→正文→标签 |
| **快手** | ≤30字 | 「老铁」风、接地气、真实感 | 2-3个标签 | 突出人设和真实感，不要太精致 |
| **B站** | ≤80字 | 分区推荐式、详细说明、互动引导 | 分区标签+话题标签 | 可加时间戳、分P说明 |
| **视频号** | ≤30字 | 微信生态风格、较正式 | 2-3个话题 | 偏朋友圈传播风格 |
| **YouTube Shorts** | ≤100字符（英文） | SEO优化、英文为主、hashtag结尾 | 3-5个#hashtag | 标题含关键词利于搜索 |

### 4.3 Prompt 设计思路

系统会为每个平台预制一套 Prompt 模板，用户可在设置中自定义。核心结构：

```
[System Prompt] 你是一个专业的短视频运营专家，擅长为{平台名}创作高互动率的内容。
你需要根据视频转写内容和画面描述，生成符合{平台名}调性的标题和描述。

平台规则：{平台规则摘要}
风格参考：{风格示例}

[User Prompt]
视频转写文本：{transcription}
视频画面描述：{visual_description}
视频时长：{duration}
视频类别（可选）：{category}

请生成：
1. 标题（2-3个备选）
2. 描述正文
3. 推荐标签（3-5个）
```

**Prompt 模板支持变量替换**：
- `{{transcription}}` — 转写文本
- `{{visual_description}}` — 画面描述
- `{{duration}}` — 视频时长
- `{{platform}}` — 目标平台名
- `{{category}}` — 视频分类

## 七、LLM 集成方案

### 7.1 支持的 LLM 及 API 信息

| 提供商 | API Endpoint | 模型 | 特点 |
|--------|-------------|------|------|
| **DeepSeek** | `https://api.deepseek.com/v1` | deepseek-chat | 性价比高，中文能力强，新模型支持多模态 |
| **豆包 (Doubao)** | `https://ark.cn-beijing.volces.com/api/v3` | doubao-1.5-pro-256k | 字节系，对抖音理解最深；支持视频直接输入（部分模型） |
| **Kimi (月之暗面)** | `https://api.moonshot.cn/v1` | moonshot-v1-8k | 长文本处理强，适合长描述 |
| **OpenAI** | `https://api.openai.com/v1` | gpt-4o | 多模态强，英文平台首选 |
| **通义千问** | `https://dashscope.aliyuncs.com/compatible-mode/v1` | qwen-max | 阿里系，中文理解好 |
| **自定义** | 用户自行填写 | 用户自定义 | 兼容 OpenAI 接口格式的皆可接入 |

### 7.2 音频转写方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Whisper API** | 准确率最高，多语言 | 需要额外 API Key |
| **DeepSeek Audio API** | 一次调用（转写+生成） | 依赖 DeepSeek |
| **豆包视频理解 API** | 直接传视频，无需分离音频 | 仅豆包生态 |
| **Web Speech API** | 免费、离线、隐私安全 | 准确率一般，仅 Chrome 支持好 |

**建议默认方案**：优先使用当前选中的 LLM 自带的音频/视频接口；如果没有，fallback 到 Whisper API。

## 八、数据流

### 8.1 生成流程

```
1. 用户上传视频 → 浏览器本地处理
2. 提取音频 → 转 base64 / Blob
3. 提取关键帧 → Canvas 截图 → base64[]
4. [可选] 用户编辑/确认转录文本
5. 构建 Prompt + 音频/帧数据 → 发送至 LLM API
6. LLM 返回结构化 JSON → 前端解析
7. 按平台渲染结果卡片
8. 用户点击复制 → Clipboard API 写入剪贴板
```

### 8.2 数据在本地处理，不上传中间服务器

```
用户的视频文件 ──▶ 浏览器本地（JS处理）──▶ 直接发送至 LLM API
                                            │
                              没有中间服务器存储或转发
```

**隐私保证**：视频和音频数据直接从用户浏览器发送到 LLM API，插件本身不经过任何第三方服务器。

## 九、增强想法（超出基本需求）

### 9.1 可在 V1 实现的
- **批量处理**：一次上传多个视频，队列生成，适合做矩阵号的用户
- **热门话题推荐**：抓取各平台当前热门话题/标签，自动融入生成内容
- **历史记录**：保存生成历史，支持搜索和复用
- **自定义文案风格**：如「激进营销风」「温和分享风」「专业测评风」等预设
- **字数实时显示**：生成结果旁显示字数统计，超限标红

### 9.2 可在后续版本考虑的
- **A/B 文案**：一个视频生成两套不同风格的文案，方便测试哪种效果好
- **竞品分析**：输入竞品视频链接，分析其文案策略，为自己的视频生成参考建议
- **多语言翻译**：同一视频生成中英日韩多语言版本，适配出海需求
- **定时发布草稿**：与平台 API 对接，直接将生成的标题描述填入发布草稿箱
- **数据分析反馈**：发布后回传播放量/互动数据，优化后续生成策略

### 9.3 产品名称建议
- **ClipCopy** — 简洁，国际化
- **剪映AI文案助手** — 如果考虑品牌化
- **一键多平台文案** — 功能直白
- **Video2Post** — 技术感
- **快剪·多平台助手** — 中文友好

## 十二、关键设计决策说明

### 为什么不在 popup 中做视频处理？
Popup 生命周期短（失去焦点即销毁），视频处理是异步耗时操作。推荐做法：
- 轻量处理（短视频 < 3 分钟）直接在 popup 中完成
- 如果用户需要长时间处理，弹出独立 tab 页（`chrome.windows.create`）

### 为什么不引入 ffmpeg.wasm？
- ffmpeg.wasm 体积约 30MB+，严重影响扩展加载速度
- 浏览器原生 `<video>` + Canvas API 已经足够提取关键帧
- 音频提取用 `AudioContext.createMediaElementSource` + `OfflineAudioContext`，但要注意 CORS 限制
- **实际情况**：浏览器环境中从 `<video>` 元素提取纯音频轨道有一定限制，V1 建议直接传视频文件给支持多模态的 LLM（如豆包、GPT-4o），或使用 Whisper API 的音频端点

### API Key 安全
- 所有 API Key 存储在 `chrome.storage.local`，仅插件自身可读取
- 不出现在前端代码仓库中
- 后续可增加加密存储（使用 chrome.storage.session 结合用户主密码）
